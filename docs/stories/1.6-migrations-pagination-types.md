## Story

**Como** developer do ProspectAI,
**Quero** que o schema do banco seja versionado, os resultados paginados e os tipos TypeScript precisos,
**Para que** o projeto escale de forma controlada e segura.

---

## Acceptance Criteria

- [ ] AC1: Schema inicial migrado para `lib/supabase/migrations/001_initial_schema.sql`
- [ ] AC2: UPDATE policies incluidas na migration
- [ ] AC3: API `/api/prospect` aceita parametros `page` e `limit` (default: page=1, limit=20)
- [ ] AC4: Response inclui `{ leads, total, page, limit, totalPages }`
- [ ] AC5: Frontend exibe paginacao (anterior/proximo) nos resultados
- [ ] AC6: Tipos `any[]` substituidos por tipos concretos em `types/index.ts`
- [ ] AC7: `regularOpeningHours`, `photos`, `reviews` tipados corretamente

---

## Scope

### In
- Migracao do schema.sql para migrations numeradas
- Paginacao server-side na API de busca
- Componente de paginacao no frontend
- Tipagem forte para campos `any`

### Out
- Cursor-based pagination (offset-based e suficiente para MVP)
- Historico de buscas com paginacao
- Infinite scroll

## Risk Assessment

| Risco | Mitigacao |
|-------|-----------|
| Paginacao no prospect route e client-side (Gemini retorna array completa) | OK para MVP — server-side pagination so faz sentido quando DB tiver historico |
| Tipagem forte pode quebrar parsing do Gemini response | Manter `any` no parse do Gemini, tipar apenas no frontend |
