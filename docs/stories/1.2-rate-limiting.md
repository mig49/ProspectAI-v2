## Story

**Como** owner do ProspectAI,
**Quero** que as APIs tenham rate limiting por usuario,
**Para que** o sistema esteja protegido contra abuso e explosao de custos do Gemini API.

---

## Acceptance Criteria

- [ ] AC1: POST `/api/prospect` limitado a 10 requests/minuto por usuario
- [ ] AC2: POST `/api/prospect/report` limitado a 20 requests/minuto por usuario
- [ ] AC3: Requests que excedem o limite recebem HTTP 429 com mensagem clara
- [ ] AC4: Response inclui headers `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- [ ] AC5: Rate limiting funciona por user_id (autenticado) ou IP (fallback)

## Risk Assessment

| Risco | Mitigacao |
|-------|-----------|
| In-memory Map reseta a cada cold start do serverless | Aceitavel para MVP — pior caso e rate limit mais permissivo, nao mais restritivo |
| Cada instancia Vercel tem Map independente (nao distribuido) | Limites generosos (10/min) compensam — abuso real requer muitos requests |
| Power users podem perceber limite de 10 buscas/min | Mensagem 429 clara em PT-BR com tempo de reset |

## Dev Notes

**Abordagem in-memory (MVP):**
```typescript
// lib/rate-limit.ts
const requests = new Map<string, { count: number; resetTime: number }>();

export function rateLimit(id: string, limit: number, windowMs: number) {
  const now = Date.now();
  const record = requests.get(id);
  if (!record || now > record.resetTime) {
    requests.set(id, { count: 1, resetTime: now + windowMs });
    return { success: true, remaining: limit - 1 };
  }
  if (record.count >= limit) {
    return { success: false, remaining: 0, reset: new Date(record.resetTime) };
  }
  record.count++;
  return { success: true, remaining: limit - record.count };
}
```

**Nota:** Em Vercel serverless, cada instancia tem seu proprio Map. Isso significa que o rate limiting nao e perfeito (distribuido), mas e suficiente para MVP.
